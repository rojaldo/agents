Introducción a n8n

n8n es una plataforma de código abierto para automatización de workflows que permite conectar aplicaciones y servicios sin escribir código complejo. Este temario cubre desde conceptos básicos hasta implementaciones avanzadas en producción.

1. MÓDULO 1: Fundamentos de n8n

1.1. 1.1 ¿Qué es n8n?

1.1.1. Definición y características principales

n8n (pronunciado "n-eight-n") es una plataforma de automatización de workflows de código abierto que permite conectar aplicaciones, servicios y APIs sin necesidad de escribir código complejo. Su nombre proviene de la palabra "nodemation" (node + automation).

Características principales:

  • Visual Workflow Builder: Editor visual basado en nodos que permite crear flujos de trabajo de manera intuitiva mediante drag-and-drop

  • Código Abierto: Licencia fair-code, permitiendo revisar y modificar el código fuente

  • Self-hosted: Control total sobre tus datos al alojar n8n en tu propia infraestructura

  • Extensible: Más de 400+ integraciones nativas y posibilidad de crear nodos personalizados

  • Flexible: Soporta JavaScript, Python y expresiones personalizadas para lógica compleja

  • REST API: API completa para gestionar workflows programáticamente

1.1.3. Ventajas de n8n

1. Código Abierto y Transparencia * Auditable: puedes revisar exactamente qué hace cada nodo * Comunidad activa: contribuciones constantes y mejoras * Sin vendor lock-in: libertad para migrar o modificar

2. Self-hosted y Control de Datos * Datos sensibles permanecen en tu infraestructura * Cumplimiento con GDPR, HIPAA y otras regulaciones * Sin límites de ejecución o transferencia de datos

3. Escalabilidad * Horizontal: múltiples workers para distribuir carga * Vertical: ajustar recursos según necesidades * Queue system: gestión eficiente de workflows concurrentes

4. Flexibilidad Técnica * Code nodes para lógica compleja en JavaScript/Python * Expresiones personalizadas con sintaxis similar a JavaScript * HTTP Request node universal para cualquier API

5. Costo-Efectivo * Gratis para instalaciones self-hosted * Sin costos por ejecución o transferencia de datos * Cloud version con precios competitivos

1.1.4. Casos de uso reales

1. Automatización de Marketing * Sincronizar leads entre CRM y herramientas de email marketing * Enviar emails personalizados basados en comportamiento del usuario * Publicar contenido automáticamente en redes sociales

2. Integración de Sistemas * ETL: Extraer datos de múltiples fuentes, transformar y cargar en data warehouse * Sincronización bidireccional entre ERP y e-commerce * Integrar sistemas legacy con aplicaciones modernas

3. DevOps y Monitoreo * Notificaciones de alertas de monitoreo en Slack/Teams * Deployment automático tras merge en repositorio * Recolección de métricas y generación de reportes

4. Procesamiento de Datos * Scraping de sitios web y almacenamiento en base de datos * Procesamiento de archivos CSV/Excel y distribución * Validación y limpieza de datos en tiempo real

5. Atención al Cliente * Crear tickets automáticamente desde emails/formularios * Enriquecer información de clientes desde múltiples fuentes * Automatizar respuestas basadas en keywords

6. E-commerce * Sincronizar inventario entre plataformas * Procesar pedidos y notificar a sistemas de fulfillment * Generar reportes de ventas y enviarlos diariamente

1.2. 1.2 Instalación y Configuración Inicial

1.2.1. Requisitos del sistema

Mínimos: * CPU: 2 cores * RAM: 2 GB * Disco: 10 GB * Node.js: v18.x o superior (para instalación local) * Docker: v20.x o superior (para instalación con contenedores)

Recomendados para producción: * CPU: 4+ cores * RAM: 4+ GB * Disco: 50+ GB (dependiendo de volumen de ejecuciones) * PostgreSQL: v12+ (para persistencia en producción)

1.2.2. Instalación con Docker

Opción 1: Docker Run (básico)

# Ejecutar n8n con SQLite (desarrollo)
docker run -it --rm \
  --name n8n \
  -p 5678:5678 \
  -v ~/.n8n:/home/node/.n8n \
  n8nio/n8n

Opción 2: Docker Compose (recomendado)

Crear archivo docker-compose.yml:

version: '3.8'

services:
  n8n:
    image: n8nio/n8n:latest
    container_name: n8n
    restart: unless-stopped
    ports:
      - "5678:5678"
    environment:
      - N8N_BASIC_AUTH_ACTIVE=true
      - N8N_BASIC_AUTH_USER=admin
      - N8N_BASIC_AUTH_PASSWORD=changeme
      - N8N_HOST=localhost
      - N8N_PORT=5678
      - N8N_PROTOCOL=http
      - WEBHOOK_URL=http://localhost:5678/
      - GENERIC_TIMEZONE=Europe/Madrid
    volumes:
      - n8n_data:/home/node/.n8n

volumes:
  n8n_data:
    driver: local

Iniciar con:

docker-compose up -d

1.2.6. Configuración básica de seguridad

1. Autenticación básica:

export N8N_BASIC_AUTH_ACTIVE=true
export N8N_BASIC_AUTH_USER=admin
export N8N_BASIC_AUTH_PASSWORD=password_muy_seguro

2. Variables de entorno importantes:

# Deshabilitar registro de nuevos usuarios
export N8N_PERSONALIZATION_ENABLED=false

# Restringir acceso a workflows
export N8N_USER_MANAGEMENT_DISABLED=false

# Configurar HTTPS
export N8N_PROTOCOL=https
export N8N_HOST=mi-dominio.com

# Webhook URL segura
export WEBHOOK_URL=https://mi-dominio.com/

3. Reverse Proxy con Nginx (recomendado para producción):

server {
    listen 80;
    server_name n8n.mi-dominio.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name n8n.mi-dominio.com;

    ssl_certificate /etc/letsencrypt/live/n8n.mi-dominio.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/n8n.mi-dominio.com/privkey.pem;

    location / {
        proxy_pass http://localhost:5678;
        proxy_set_header Connection '';
        proxy_http_version 1.1;
        chunked_transfer_encoding off;
        proxy_buffering off;
        proxy_cache off;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

4. Firewall básico (UFW en Ubuntu):

# Permitir solo SSH y HTTP/HTTPS
sudo ufw allow OpenSSH
sudo ufw allow 'Nginx Full'
sudo ufw enable

# Bloquear acceso directo al puerto 5678
sudo ufw deny 5678

1.4. 1.4 Conceptos Fundamentales

1.4.1. Workflows (flujos de trabajo)

Un workflow es una secuencia de pasos automatizados que se ejecutan en orden para completar una tarea específica.

Componentes de un workflow: * Nombre: Identificador único y descriptivo * Descripción: Documentación del propósito * Nodos: Elementos individuales que realizan acciones * Conexiones: Enlaces que determinan el flujo de datos * Trigger: Punto de inicio del workflow * Settings: Configuración global del workflow

Ciclo de vida de un workflow:

┌─────────┐     ┌──────────┐     ┌──────────┐     ┌─────────┐
│ Trigger │────>│ Process  │────>│ Process  │────>│  End    │
│ (Inicio)│     │  Data    │     │  Data    │     │         │
└─────────┘     └──────────┘     └──────────┘     └─────────┘
     ▲                                                   │
     │                                                   │
     └───────────────── Loop/Retry ─────────────────────┘

Estados de un workflow: * Inactive: No está ejecutándose * Active: Activado y esperando trigger * Running: En ejecución actual * Error: Falló durante la ejecución * Success: Completado exitosamente

Ejemplo básico - Estructura JSON: Ver archivo completo: ./ejemplos/n8n/01-workflow-basico.json

1.4.2. Nodos y conexiones

Nodos: Los nodos son los bloques de construcción de n8n. Cada nodo realiza una tarea específica.

Tipos de nodos: * Trigger nodes: Inician el workflow * Regular nodes: Procesan o transforman datos * Conditional nodes: Toman decisiones (IF, Switch) * Merge nodes: Combinan datos de múltiples fuentes

Anatomía de un nodo:

┌───────────────────────────────┐
│  ⚡ HTTP Request              │  ← Nombre del nodo
│  ────────────────────────     │
│  URL: api.example.com         │  ← Configuración
│  Method: GET                  │
│  ────────────────────────     │
│  ● Input  │  Output ●        │  ← Puntos de conexión
└───────────────────────────────┘

Conexiones: Las conexiones determinan cómo fluyen los datos entre nodos.

Conexión simple:
Node A ──────> Node B

Conexión condicional (IF):
              ┌──> Node B (true)
Node A ───────┤
              └──> Node C (false)

Conexión múltiple (múltiples outputs):
         ┌──> Node B
Node A ──┼──> Node C
         └──> Node D

Flujo de datos: Los datos se pasan de nodo a nodo como un array de items:

[
  {
    "json": {
      "name": "Item 1",
      "value": 100
    }
  },
  {
    "json": {
      "name": "Item 2",
      "value": 200
    }
  }
]

1.4.3. Triggers (disparadores)

Los triggers son nodos especiales que inician la ejecución de un workflow.

Tipos de triggers:

1. Manual Trigger * Se ejecuta manualmente desde la interfaz * Útil para testing y desarrollo

2. Webhook Trigger * Se ejecuta cuando recibe una petición HTTP * Proporciona una URL única * Soporta GET, POST, PUT, DELETE

Diagrama Webhook:
┌──────────────┐      HTTP Request       ┌─────────────┐
│   Cliente    │─────────────────────────>│  Webhook    │
│  (Frontend)  │<─────────────────────────│  (n8n)      │
└──────────────┘      HTTP Response       └─────────────┘
                                                 │
                                                 ▼
                                          ┌─────────────┐
                                          │  Workflow   │
                                          │  Execution  │
                                          └─────────────┘

3. Schedule Trigger (Cron) * Se ejecuta en intervalos de tiempo específicos * Usa expresiones cron * Ejemplo: 0 9 * * * (todos los días a las 9 AM)

4. Email Trigger * Se ejecuta cuando llega un email * Monitorea una cuenta de email (IMAP)

5. Form Trigger * Crea un formulario web * Se ejecuta cuando se envía el formulario

Ejemplo de trigger webhook: Ver archivo: ./ejemplos/n8n/02-webhook-trigger.json

1.4.4. Acciones

Las acciones son nodos que realizan operaciones específicas.

Categorías principales:

1. Comunicación: * Email Send: Enviar emails * Slack: Enviar mensajes a Slack * Telegram: Enviar mensajes a Telegram * Discord: Enviar mensajes a Discord

2. Datos: * HTTP Request: Llamar a APIs * Database: Operaciones CRUD en BD * Google Sheets: Leer/escribir hojas * File Operations: Leer/escribir archivos

3. Lógica: * IF: Condicionales * Switch: Múltiples condiciones * Merge: Combinar datos * Split: Dividir items

4. Transformación: * Set: Establecer valores * Code: Ejecutar JavaScript/Python * Function: Transformaciones simples * Item Lists: Manipular arrays

Diagrama de flujo con acciones:

┌─────────┐    ┌──────────┐    ┌─────────┐    ┌─────────┐
│ Webhook │───>│   HTTP   │───>│   IF    │───>│  Email  │
│ Trigger │    │ Request  │    │         │    │  Send   │
└─────────┘    └──────────┘    └────┬────┘    └─────────┘
                                     │
                                     ▼
                                ┌─────────┐
                                │  Slack  │
                                │  Send   │
                                └─────────┘

1.4.5. Datos y tipos de datos

Estructura de datos en n8n:

Todos los datos en n8n se pasan entre nodos como un array de items:

[
  {
    "json": { ... },        // Datos principales (JSON)
    "binary": { ... },      // Datos binarios (opcional)
    "pairedItem": { ... }   // Referencia al item origen
  }
]

Tipos de datos soportados:

1. JSON (más común):

{
  "json": {
    "id": 123,
    "name": "John Doe",
    "email": "john@example.com",
    "active": true,
    "tags": ["customer", "premium"],
    "metadata": {
      "created": "2025-01-15",
      "updated": "2025-01-15"
    }
  }
}

2. Binary (archivos, imágenes):

{
  "json": {
    "fileName": "document.pdf"
  },
  "binary": {
    "data": {
      "data": "base64_encoded_data...",
      "mimeType": "application/pdf",
      "fileName": "document.pdf",
      "fileSize": 1024000
    }
  }
}

3. Múltiples items:

[
  { "json": { "id": 1, "name": "Item 1" } },
  { "json": { "id": 2, "name": "Item 2" } },
  { "json": { "id": 3, "name": "Item 3" } }
]

Acceso a datos:

En expresiones, puedes acceder a datos usando: * $json: Datos del item actual * $input: Datos de entrada del nodo * $node: Datos de un nodo específico * $binary: Datos binarios

Ejemplos:

// Acceder a un campo
{{ $json.email }}

// Acceder a campo anidado
{{ $json.user.address.city }}

// Acceder a array
{{ $json.items[0].name }}

// Datos de nodo anterior específico
{{ $node["HTTP Request"].json.status }}

Las expresiones permiten usar datos dinámicos en lugar de valores estáticos.

Funciones integradas:

String:

{{ $json.name.toUpperCase() }}

{{ $json.message.substring(0, 10) }}
{{ $json.name.split(' ')[0] }}

Array:

{{ $json.numbers.filter(x => x > 10) }}

2.3.4. Trigger: Schedule (Cron)

El Schedule Trigger ejecuta el workflow en intervalos regulares según una expresión cron.

Expresiones Cron básicas:

Formato:
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ Minuto   │ Hora     │ Día      │ Mes      │ Día Sem. │
│ (0-59)   │ (0-23)   │ (1-31)   │ (1-12)   │ (0-6)    │
│ *        │ *        │ *        │ *        │ *        │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│   30     │   9      │   *      │   *      │   *      │
│ (Minuto) │ (Hora)   │ (Todos)  │ (Todos)  │ (Todos)  │
└──────────┴──────────┴──────────┴──────────┴──────────┘

Ejemplos comunes:

"0 9 * * *"
= Todos los días a las 9:00 AM
= Minuto 0, Hora 9

"*/15 * * * *"
= Cada 15 minutos
= * = todos, /15 = cada 15

"0 0 * * 0"
= Todos los domingos a las 00:00
= Día de semana 0 = domingo

"0 9-17 * * 1-5"
= De lunes a viernes, cada hora 9-17 (9 AM a 5 PM)

"0 0 1 * *"
= El primer día de cada mes a las 00:00

"30 23 * * *"
= Todos los días a las 23:30 (11:30 PM)

"0 */6 * * *"
= Cada 6 horas (00:00, 06:00, 12:00, 18:00)

Herramienta de generador Cron:

https://crontab.guru/ es una excelente herramienta para generar y validar expresiones cron.

Configuración en n8n:

│ ○ Every custom interval          │
│                                  │
│ Cron Expression:                 │
│ [0 9 * * * ]                    │
│  ↑ = Cada día a las 9:00 AM     │
│                                  │
│ Time Zone:                       │
│ [Europe/Madrid           ▼]      │
│                                  │
│ [Copy Cron]                      │
│ [Cron Guru Link] 🔗              │
│                                  │
└──────────────────────────────────┘

3. MÓDULO 3: Nodos y Conectores

Los nodos son los bloques básicos de construcción en n8n, similares a LEGO: cada pieza tiene un propósito específico y se conectan entre sí para crear flujos complejos. En este módulo aprenderás todos los tipos de nodos disponibles y cómo utilizarlos.

3.1. 3.1 Tipos de Nodos

3.1.1. ¿Qué es un Nodo?

Un nodo es una unidad de trabajo que realiza una acción específica: - Recibe datos de entrada (de otros nodos o del usuario) - Procesa esos datos según su configuración - Emite datos de salida hacia otros nodos

La belleza de n8n está en que puedes conectar nodos sin escribir código (aunque también puedes hacerlo si lo necesitas).

3.1.2. 3.1.1 Nodos de Entrada (Triggers)

Los triggers son nodos que inician un workflow. Sin un trigger, el workflow nunca se ejecuta.

Tipo Descripción Ejemplo

Formulario web → n8n

Schedule

Ejecuta en horarios específicos (cron)

Cada día a las 9 AM

Manual

Se ejecuta al hacer clic en "Execute Workflow"

Desarrollo y testing

File

Se activa cuando se sube un archivo

Procesamiento de documentos

Se activa cuando llega un email

Procesamiento de mensajes

Database

Polling: verifica BD regularmente

Sincronización de datos

Analogía: Si un workflow fuese una película, los triggers serían el evento que hace que empiece (ej: "cuando el usuario presiona un botón").

Ejemplo práctico - Webhook:

2. n8n genera URL: https://hook.n8n.io/webhook/abc123
3. Configurar formulario web para enviar datos a esa URL
4. Cuando alguien envía el formulario → webhook dispara el workflow

3.4. 3.4 Nodos de Lógica

3.4.1. 3.4.1 IF Node

Condicionales simples: Si X es verdad, ejecuta rama A. Si falso, rama B.

Sintaxis:

True branch: [Slack notification]
False branch: [Log in database]

Si {{$node.HTTPRequest.data.statusCode}} === 200
  → Guardar en base de datos
Sino
  → Enviar alerta a Slack

3.6. 3.6 Autenticación

La autenticación permite que n8n acceda a APIs de forma segura.

3.6.1. Tipos de autenticación:

1. API Key

GET /api/endpoint?api_key=sk_1234567890

  • Simple

  • Guardar key en credenciales de n8n

  • Riesgo: si se expone la key, alguien puede usarla

2. OAuth 2.0 (estándar seguro)

1. Usuario hace click en "Connect Google"
2. Se abre ventana de login de Google
3. Usuario autoriza n8n
4. Google envía token a n8n
5. n8n guarda token encriptado

  • Seguro (n8n nunca ve la contraseña)

  • Permite permiso granular ("solo lectura en Sheets")

  • Se usa con: Google, GitHub, Slack, Microsoft, Stripe

3. Basic Auth

Header: Authorization: Basic base64(user:password)

  • Simple pero menos seguro

  • Usar solo en conexiones HTTPS/SSL

4. Bearer Token

Header: Authorization: Bearer token_aqui

  • Usado por muchas APIs modernas (ej: OpenAI, HuggingFace)

5. Custom Headers

Header: X-API-Key: sk_1234567890
Header: X-Custom-Auth: valor

  • Cuando la API tiene requerimientos especiales

Best Practices: - Nunca poner keys en hardcode en nodos - Siempre usar credenciales de n8n (encriptadas) - Rotar keys regularmente - Usar secrets management en producción (env variables) - Auditar quién tiene acceso a qué credenciales

4. MÓDULO 4: Expresiones y Variables

Las expresiones son el "pegamento" que permite que los datos fluyan dinámicamente a través de los nodos. Sin expresiones, n8n solo podría trabajar con valores estáticos.

4.1. 4.1 Sintaxis de Expresiones

4.1.1. ¿Qué es una expresión?

Una expresión es código que accede, transforma y usa datos de nodos anteriores.

Sintaxis básica:

{{$node.nombreNodo.data.campo}}

Desglose: - {{ y }} → Delimitador de expresión - $node → Especial: acceso a datos de nodos - nombreNodo → Nombre del nodo anterior - .data → Los datos output del nodo - .campo → Campo específico

Ejemplo real:

Si tengo un nodo "HTTP Request" que devuelve:
{
  "status": 200,
  "user": { "name": "Juan", "email": "juan@email.com" }
}

Puedo acceder a:
{{$node.HTTPRequest.data.status}}        → 200
{{$node.HTTPRequest.data.user.name}}    → "Juan"
{{$node.HTTPRequest.data.user.email}}   → "juan@email.com"

4.1.2. 4.1.1 Acceder a Datos de Nodos Anteriores

$node.NombreNodo.data.campo
$node.NombreNodo.data.campo[0]          # Primer item
$node.NombreNodo.data[0].campo          # Arrays
$node.NombreNodo.data.objeto.propiedad  # Objetos anidados

Tabla de operadores:

Operador Uso Ejemplo

.

Acceder a propiedades

{{$node.HTTP.data.user.name}}

[]

Acceder a arrays/índices

{{$node.HTTP.data.items[0]}}

?

Safe navigation (no falla si undefined)

{{$node.HTTP.data?.user?.name}}

||

OR: valor por defecto

{{$node.data.email || "sin-email@example.com"}}

Manejar 429 Too Many Requests:

// Implementar retry con backoff exponencial
async function fetchConRetry(url, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await fetch(url);
      if (response.status === 429) {
        const delay = Math.pow(2, i) * 1000; // Backoff exponencial
        await new Promise(resolve => setTimeout(resolve, delay));
        continue;
      }
      return response;
    } catch (error) {
      if (i === maxRetries - 1) throw error;
    }
  }
}
